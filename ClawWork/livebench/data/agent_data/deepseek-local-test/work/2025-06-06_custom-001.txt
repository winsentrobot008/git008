# REST API Design Document - Mobile Application

## Overview
This document outlines a comprehensive REST API design for a mobile application requiring user authentication, data synchronization, and real-time notifications.

## 1. Authentication Flow

### 1.1 Registration (POST /api/v1/auth/register)
- **Request Body:** `{ "email": "string", "password": "string", "full_name": "string", "device_id": "string" }`
- **Response:** `{ "user_id": "uuid", "access_token": "jwt", "refresh_token": "jwt", "expires_in": 3600 }`
- **Validations:** Email format verification, password strength (min 8 chars, uppercase, lowercase, digit, special char), unique email constraint
- **Rate Limiting:** 5 requests per minute per IP

### 1.2 Login (POST /api/v1/auth/login)
- **Request Body:** `{ "email": "string", "password": "string", "device_id": "string" }`
- **Response:** `{ "access_token": "jwt", "refresh_token": "jwt", "expires_in": 3600, "user": { "id": "uuid", "email": "string", "full_name": "string" } }`
- **Rate Limiting:** 10 requests per minute per IP; lockout after 5 failed attempts (15 min cooldown)

### 1.3 Token Refresh (POST /api/v1/auth/refresh)
- **Request Body:** `{ "refresh_token": "jwt" }`
- **Response:** `{ "access_token": "jwt", "refresh_token": "jwt", "expires_in": 3600 }`
- **Logic:** Validates refresh token, checks revocation list, issues new token pair
- **Refresh Token Rotation:** Old refresh token is invalidated on each refresh

### 1.4 Logout (POST /api/v1/auth/logout)
- **Headers:** `Authorization: Bearer <access_token>`
- **Request Body:** `{ "refresh_token": "jwt", "device_id": "string" }`
- **Response:** 204 No Content
- **Logic:** Adds tokens to revocation list, clears device session

### 1.5 Forgot Password (POST /api/v1/auth/forgot-password)
- **Request Body:** `{ "email": "string" }`
- **Response:** 202 Accepted (always returns success to prevent email enumeration)
- **Logic:** Generates 6-digit OTP, stores with 15-min expiry, sends via email

### 1.6 Reset Password (POST /api/v1/auth/reset-password)
- **Request Body:** `{ "email": "string", "otp": "string", "new_password": "string" }`
- **Response:** 200 OK on success, 400 on invalid/expired OTP

### 1.7 OAuth2 Social Login (POST /api/v1/auth/oauth/{provider})
- **Provider:** google, apple, facebook
- **Request Body:** `{ "id_token": "string", "device_id": "string" }`
- **Flow:** Verifies token with provider, finds or creates user, returns JWT pair
- **Account Linking:** If email matches existing account, prompts for linking

## 2. JWT Token Structure

### Access Token
```json
{
  "sub": "user_uuid",
  "email": "user@example.com",
  "role": "user",
  "device_id": "device_uuid",
  "iat": 1717660800,
  "exp": 1717664400,
  "jti": "unique_token_id"
}
```

### Refresh Token
```json
{
  "sub": "user_uuid",
  "type": "refresh",
  "jti": "unique_token_id",
  "iat": 1717660800,
  "exp": 1717747200
}
```

- **Access Token TTL:** 1 hour (3600 seconds)
- **Refresh Token TTL:** 30 days (2,592,000 seconds)
- **Signing Algorithm:** RS256 (asymmetric key pair)
- **Revocation Strategy:** Redis-based blacklist with TTL matching token expiry

## 3. Core API Endpoints

### 3.1 User Profile
- **GET /api/v1/users/me** — Get authenticated user profile
- **PATCH /api/v1/users/me** — Update profile `{ "full_name": "string", "avatar_url": "string", "phone": "string", "timezone": "string" }`
- **DELETE /api/v1/users/me** — Account deletion (soft delete with 30-day grace period)
- **GET /api/v1/users/me/devices** — List authorized devices
- **DELETE /api/v1/users/me/devices/{device_id}** — Revoke device access

### 3.2 Data Synchronization
- **GET /api/v1/sync/status** — `{ "last_synced_at": "ISO8601", "pending_changes": 42, "conflicts": 3 }`
- **POST /api/v1/sync/pull** — Pull changes since timestamp
  - **Request:** `{ "last_sync_timestamp": "ISO8601", "entity_types": ["tasks", "notes", "contacts"], "batch_size": 100 }`
  - **Response:** `{ "changes": [ { "entity_type": "string", "entity_id": "uuid", "action": "create|update|delete", "data": {}, "version": 5, "timestamp": "ISO8601" } ], "has_more": false, "new_sync_token": "string" }`
- **POST /api/v1/sync/push** — Push local changes to server
  - **Request:** `{ "changes": [ { "entity_type": "string", "entity_id": "uuid", "action": "create|update|delete", "data": {}, "client_version": 3, "device_id": "string" } ], "sync_token": "string" }`
  - **Response:** `{ "accepted": [...], "conflicts": [ { "entity_id": "uuid", "server_version": 5, "server_data": {}, "client_version": 3, "client_data": {} } ], "new_sync_token": "string" }`
- **Conflict Resolution Strategy:** Last-Write-Wins (LWW) with server timestamp precedence; conflicts returned to client for manual resolution option

### 3.3 Real-time Notifications
- **WebSocket Endpoint:** `wss://api.example.com/ws/v1/notifications?token={access_token}`
- **WebSocket Events:**
  - `notification.new` — `{ "id": "uuid", "type": "message|alert|sync_complete", "title": "string", "body": "string", "data": {}, "created_at": "ISO8601" }`
  - `sync.conflict_detected` — `{ "entity_id": "uuid", "conflict_data": {} }`
  - `presence.update` — `{ "user_id": "uuid", "status": "online|offline|away", "last_seen": "ISO8601" }`

### 3.4 REST Notification Endpoints
- **GET /api/v1/notifications** — List notifications (paginated) `?page=1&per_page=20&type=message&unread_only=true`
- **PATCH /api/v1/notifications/{id}/read** — Mark as read
- **POST /api/v1/notifications/read-all** — Mark all as read
- **DELETE /api/v1/notifications/{id}** — Delete notification
- **GET /api/v1/notifications/settings** — Get notification preferences
- **PATCH /api/v1/notifications/settings** — Update preferences `{ "push_enabled": true, "email_digest": "daily|weekly|never", "quiet_hours": { "start": "22:00", "end": "07:00", "timezone": "America/New_York" } }`

## 5. Error Handling Strategy

### Standard Error Response Format
```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "The request data is invalid.",
    "details": [
      {
        "field": "email",
        "message": "Must be a valid email address",
        "code": "INVALID_FORMAT"
      },
      {
        "field": "password",
        "message": "Must be at least 8 characters",
        "code": "TOO_SHORT"
      }
    ],
    "request_id": "req_uuid",
    "timestamp": "ISO8601",
    "help_url": "https://docs.example.com/errors/validation_error"
  }
}
```

### HTTP Status Codes
| Code | Description | Usage |
|------|-------------|-------|
| 200 | OK | Successful GET, PATCH |
| 201 | Created | Successful POST (resource created) |
| 204 | No Content | Successful DELETE, Logout |
| 400 | Bad Request | Validation errors, malformed requests |
| 401 | Unauthorized | Missing/invalid/expired access token |
| 403 | Forbidden | Valid token but insufficient permissions |
| 404 | Not Found | Resource doesn't exist |
| 409 | Conflict | Sync conflict, duplicate resource |
| 422 | Unprocessable Entity | Business rule violations |
| 429 | Too Many Requests | Rate limit exceeded |
| 500 | Internal Server Error | Unexpected server failures |

### Error Codes
- **AUTH_INVALID_CREDENTIALS** — Wrong email/password
- **AUTH_TOKEN_EXPIRED** — Access token expired
- **AUTH_TOKEN_REVOKED** — Token has been revoked
- **AUTH_REFRESH_INVALID** — Refresh token invalid or rotated
- **VALIDATION_ERROR** — Request body validation failed
- **RESOURCE_NOT_FOUND** — Requested resource doesn't exist
- **RESOURCE_CONFLICT** — Version conflict during sync
- **RATE_LIMIT_EXCEEDED** — Too many requests
- **INTERNAL_ERROR** — Server-side failure
- **SYNC_CONFLICT** — Data synchronization conflict detected

### Rate Limiting Strategy
- **Authentication endpoints:** 10 req/min per IP
- **Sync endpoints:** 30 req/min per user
- **General endpoints:** 100 req/min per user
- **Headers returned:** `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`
- **Backoff strategy:** Exponential backoff with jitter recommended for clients

### Security Headers
```
X-Content-Type-Options: nosniff
X-Frame-Options: DENY
Strict-Transport-Security: max-age=31536000; includeSubDomains
Content-Security-Policy: default-src 'self'
X-XSS-Protection: 1; mode=block
```

## 6. Pagination Strategy
- **Cursor-based pagination** for real-time feeds (notifications, sync changes)
  - Request: `?cursor=eyJsYXN0X2lkIjogIjEyMyJ9&limit=20`
  - Response includes: `{ "data": [...], "next_cursor": "string", "has_more": true }`
- **Offset-based pagination** for admin/list views
  - Request: `?page=1&per_page=25`
  - Response includes: `{ "data": [...], "pagination": { "page": 1, "per_page": 25, "total": 142, "total_pages": 6 } }`

## 7. API Versioning
- **Strategy:** URL path versioning (`/api/v1/`)
- **Deprecation Policy:** Minor versions backward-compatible; major versions deprecated 6 months in advance with Sunset header
- **Sunset Header:** `Sunset: Sat, 06 Dec 2025 00:00:00 GMT`

## 8. Caching Strategy
- **ETags** for user profile and settings endpoints
- **Cache-Control: private, max-age=60** for user-specific data
- **Cache-Control: public, max-age=300** for static reference data
- **Last-Modified** headers for sync endpoints

## 9. WebSocket Reconnection Strategy
- **Exponential backoff:** 1s → 2s → 4s → 8s → max 30s
- **Max retries:** 10 attempts before falling back to polling
- **Heartbeat:** Client sends ping every 30s; server responds pong
- **Connection recovery:** Server buffers last 50 events per user for 2 minutes after disconnect
